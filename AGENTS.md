# ELITE — правила кода

Космический сим: честная физика, аркадное управление, PS1-эстетика.
Цель архитектуры — чтобы симуляция однажды запустилась на сервере **без единой правки**.

## Стек

- **Vite + React 19 + TypeScript + Tailwind v4.** Не Next.js: игра — клиентский WebGL-остров,
  SSR ей не нужен. Next добавим, когда понадобится сайт вокруг игры (лендинг, аккаунты, рекорды).
- **npm workspaces.** `packages/sim` — чистая симуляция, `apps/web` — клиент.
  Сетевой сервер (`apps/server`) появится как третий пакет и импортирует тот же `sim`.
- **three.js + @react-three/fiber** — только в `apps/web`.

## Слои и направление зависимостей

```
config  ←  core  ←  domain  ←  render / ui  ←  app
```

Импорт разрешён **только справа налево**. Стрелка вверх — ошибка, а не стиль.

| Слой       | Что там                                              | Что запрещено                          |
|------------|------------------------------------------------------|----------------------------------------|
| `config`   | Числа и настройки. Единственный источник правды.      | Логика.                                |
| `core`     | Чистая математика, RNG, пересечения.                 | Знание об игре.                        |
| `domain`   | Правила игры: полёт, бой, ИИ, мир, шаг симуляции.    | three-сцена, React, DOM, глобалы.      |
| `render`   | three.js: материалы, геометрия, компоненты сцены.    | Правила игры.                          |
| `ui`       | React-оверлеи: HUD, карта, меню.                     | Правила игры.                          |
| `app`      | Композиция: Canvas, петля, привязка ввода.           | Своя логика сверх склейки.             |

## Домен обязан оставаться чистым

Это не эстетика — это то, что позволит запустить его на сервере.

- **Нельзя** `Object3D`, `Mesh`, `Scene`, материалы, `useFrame`, React, `document`, `window`.
- **Можно** `Vector3` / `Quaternion` / `Euler` из three: это чистая математика, тот же пакет,
  что уже в сборке. Единственное послабление, и оно осознанное.
- **Нельзя `Math.random()`.** Только seeded RNG, переданный аргументом (`Rng`).
  Недетерминированная симуляция не синхронизируется по сети — ни лок-степом, ни откатом.
- **Нельзя `Date.now()` / `performance.now()`.** Время приходит параметром (`time`, `dt`).
- **Нельзя читать ввод.** Домен видит только `ShipControls` — заполненные структуры.

## Игрок и бот неотличимы для физики

`Controller` — единственный интерфейс между «кто решает» и «что летит».
`PlayerController` читает мышь, `AiController` думает — оба пишут в один и тот же `ShipControls`.

- Симуляция **не импортирует** ни ИИ, ни ввод. Она вызывает `Controller` (DIP).
- Врагу нельзя дать угловую скорость выше игроковой. Он выигрывает решением, не привилегией.
- Слабости ИИ моделируются честно: задержка реакции, разброс, инерция — не понижением урона.

## Состояние игры — не состояние React

- Мир живёт в обычном мутируемом объекте. React **никогда** не перерисовывается ради кадра.
- HUD читает мир в `useFrame` и пишет напрямую в `ref.current.style` / `textContent`.
  Никаких `setState` в игровой петле — это мгновенная смерть частоты кадров.
- Tailwind — для статики: панели, меню, хром. Живые элементы двигаются через `style`, не через классы.

## Петля

- **Фиксированный шаг** симуляции (`FIXED_DT`), переменный шаг рендера. Иначе поведение
  корабля зависит от герцовки монитора, а сеть невозможна в принципе.
- Накопитель ограничен сверху (`MAX_DT`): свёрнутая вкладка не должна телепортировать мир.

## Горячий путь без аллокаций

Интегратор зовётся сотни раз за кадр. Каждый `new Vector3()` там — мусор для GC.

- Переиспользуемые векторы на уровне модуля, префикс `_`: `const _fwd = new Vector3()`.
- Такой вектор **никогда не возвращается наружу** — только `.copy()` в `out`-параметр.
- Функции, которым нужен результат-вектор, принимают `out` последним аргументом.

## Данные вместо ветвлений (OCP)

- Новый тип корабля — это новый `ShipTuning` и новая фабрика геометрии. **Не** новый `if` в `sim`.
- Новая система — это новый `SystemDef`. Не правка `createWorld`.
- Если добавление контента требует трогать симуляцию — архитектура сломалась.

## Компоненты

- **Нет супер-компонентов.** Компонент либо рисует одну вещь, либо оркестрирует — не оба сразу.
- Компонент сцены владеет одной визуальной сущностью (`Starfield`, `Tracers`, `AsteroidField`).
- Дробим по **причине измениться**, а не по числу строк. Файл на одну функцию — тоже плохо.

## Типы и имена

- `strict: true`, `noUncheckedIndexedAccess`. `any` — запрещён; не знаешь тип — `unknown` и сузь.
- В домене имена доменные: `hull`, `shield`, `throttle`, `bank`. Не `mesh`, не `props`.
- Единицы измерения — в комментарии к константе. Метры, секунды, радианы, тонны.

## Визуальный стиль

Аккуратная низкополигональная графика. **Не эмуляция PS1.**

- Дрожание вершин, аффинные текстуры и прочие артефакты железа 1994 года — это были
  баги, а не стиль. Не воспроизводим.
- Гранёность — следствие малого числа полигонов и `flatShading`, а не постобработки.
- Покраска по вершинам вместо текстур. Палитра узкая, цвета плоские.
- Свет: жёсткий направленный от звезды + слабая заливка, чтобы теневая сторона
  не была чёрной дырой. Терминатор на планетах получается сам.
- Разрешение — одна константа в `render/config.ts`. Низкое разрешение даёт крупный
  пиксель само по себе; подделывать его нечем и незачем.
- Логарифмический буфер глубины обязателен: звезда в 1200 км и корабль в 12 м
  не уживаются в одном кадре с обычным.

## Производительность рендера

Цель — стабильные 60 кадров. Дешевле не делать работу, чем делать её быстро.

- **Один вызов отрисовки на класс объектов.** Астероиды, контейнеры, ракеты, враги —
  `InstancedMesh`. 260 астероидов = 1 draw call, а не 260.
- **Буферы выделяются один раз.** Трассы, пыль, взрывы — заранее выделенный
  `LineSegments`/`InstancedMesh` фиксированного размера; за кадр меняются только данные
  и `drawRange`/`count`. Никаких `new BufferGeometry()` в кадре.
- **Ноль аллокаций в `useFrame`.** Матрицы собираются одним переиспользуемым `Object3D`,
  векторы — модульные `_scratch`. `new Vector3()` в кадре — утечка в GC.
- **Геометрии и материалы создаются один раз** на модуль, не на компонент.
- **React не участвует в кадре.** Ни `setState`, ни новые пропсы.
- Симуляция уже с фиксированным шагом и ограниченным накопителем — не трогать.

## HUD

HUD рисуется на **отдельном 2D-канвасе того же внутреннего разрешения**, что и 3D,
и растягивается тем же множителем. Не в DOM.

Причина не стилистическая: DOM пикселизовать нельзя. `transform: scale(4)` не увеличивает
пиксели — браузер заново растеризует текст и линии в полном разрешении экрана. Вектор
всегда рисуется в физические пиксели устройства. Отсюда два следствия:

- Пикселизуется вместе с 3D только растр, значит HUD — растр.
- Общая пиксельная сетка с 3D: прицел не «плавает» на полпикселя относительно кораблей.

HUD не является React-деревом. Это императивная отрисовка в кадре: ноль перерисовок,
ноль рефов. При внутренних 320 пикселях в строку влезает ~40 символов — HUD должен быть скупым.

## Тесты

`npm test` — Vitest в `packages/sim`. Тесты лежат рядом с кодом: `*.test.ts`.

Домен тестируется без всякого рендера — в этом и был смысл его чистоты.
Если для теста нужен браузер, значит логика утекла не в тот слой.

**Что тестировать.** Не «покрытие», а инварианты и регрессии:

- **Детерминизм.** Одно зерно — одна галактика. Сломается — не будет ни сохранений, ни сети.
- **Инварианты мира.** Обитаемая система имеет столицу; станция бывает только у обитаемого мира.
- **Физические свойства, а не числа.** «Тяжёлый щит уменьшает угловое ускорение» переживёт
  перебалансировку, а `expect(pitchAccel).toBe(2.65)` сломается от первой же правки конфига.
- **Каждый пойманный баг — тест.** С комментарием, ПОЧЕМУ так, а не иначе. Уже записаны:
  упреждение промахивается мгновенным лучом; звезда не тормозит крейсер с другого конца системы.

**Числа не подгоняем под тест.** Если тест требует магической константы — тестируется не то.

**Вероятность живёт в секундах, не в шагах.** `if (rng() < 0.004)` внутри шага физики — это
не «редко», это 0.48 события в секунду при 120 Гц и 0.96 при 240. Темп обязан задаваться
перезарядом в секундах, а решение приниматься в такте размышления. Иначе поведение
симуляции меняется от частоты шага, и сеть рассыпается вместе с балансом.

**Баланс меряется, а не угадывается.** Прежде чем крутить урон, замерь, где теряется
полезное действие. Замер `scratch/balance.ts` показал: с приклеенным к цели прицелом пират
гибнет за 0.4 с, а тот же бот, рулящий сам, убивает его 50 секунд. Виновато было не оружие,
а слежение — и правка урона только сломала бы то, что работало.

**Считай, прежде чем изобретать механику.** Ракета доворачивает вектор скорости со скоростью
`turnRate`, значит её боковое ускорение равно v·ω: 420 м/с × 1.25 рад/с = 525 м/с², полсотни g.
Корабль выдаёт шестнадцать. От такой ракеты нельзя уйти НИКАКИМ манёвром, и «бочка» этого не
изменит — `scratch/barrel.ts` перебрал тягу и разворот и не нашёл ни одной пары. Промах даёт
только срыв наведения: угловая скорость линии визирования равна v⊥/d и вблизи превышает предел
слежения головки. Механика обязана следовать из чисел, а не наоборот.

**`scratch/` — не тесты.** Это исследовательские прогоны (бой без рендера, перебор параметров).
Они печатают числа для глаз. Как только число становится важным — оно переезжает в `*.test.ts`.

## Комментарии

Объясняют **почему** и какое ограничение соблюдается — не то, что и так видно в коде.

```ts
// Плохо: увеличиваем угол на dt
// Хорошо: угловая скорость задана в СВЯЗАННЫХ осях, поэтому приращение умножается справа.
```
