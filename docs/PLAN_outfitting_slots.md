# План: категориальные слоты снаряжения и экран верфи

Договорённая переработка модели снаряжения (обсуждено 2026-07-13). Цель — сделать
слоты категориальными, экран верфи наглядным, а установку — по классу корпуса.

## Согласованная модель

Каждая **категория = один слот** с ёмкостью (сколько единиц вмещает). В экране —
одна ячейка на категорию, одинакового размера. Пусто → тускло; есть хоть одна
единица → ярко, внутри подпись что стоит и «установлено / максимум».

| Слот | Ёмкость | Содержимое |
|---|---|---|
| Двигатель | 1 | двигатель |
| Маневровые (RCS) | 1 | маневровые |
| Щит | 1 | щит |
| Гиперпривод | 1 | привод |
| **Лазеры** | 1–3 | лазеры, НЕ различаются (просто счёт) |
| **Ракеты** | 1 | ОДИН тип, до макс. числа (боезапас) |
| **Броня** | 1–3 | плиты брони |
| **Груз** | 1–2 | контейнеры; подпись «M/N, всего Xт» |
| **Доп. оборуд. (аукс)** | 1–2, у дорогих 4 | ECM / клоак / бомба / скуп / миелофон / дрон / … |

## Решения (зафиксированы)

1. **Класс корпуса 1–4** ограничивает класс ЛЮБОГО оборудования на нём. Заменяет
   `maxClass` у каждого слота. **У каждого корпуса свой класс** (не все одинаковые).
   **Груз — бесклассовый, исключение** (уже сделано). Аукс-модули класс имеют СВОЙ,
   индивидуальный (не бесклассовые!), и гейтятся классом корпуса, как всё прочее.
2. **Миелофон → аукс.** Расти может ЛЮБОЙ корпус со свободной аукс-ячейкой, не только
   «Аврора». Отменяет прежнее «право на масштаб — от отдельного слота Авроры».
3. **ECM / энергобомба / топливный скуп → снаряжаемые аукс-модули.** Сейчас это
   врождённые способности; станут модулями (без модуля — не пользуешься).
4. **Сейвы лоадаута сбрасываем** (ранняя разработка). Бампнуть версию схемы сейва.
5. **Каждое аукс-устройство — на своей РОДНОЙ клавише** (ECM/бомба/клоак/… как
   сейчас). Единую E отменили: до 3 устройств → 3 привычные клавиши; «1-2-3» путают,
   родные бинды — нет. Бинды и их описание НЕ трогаем/восстанавливаем.
6. **Аукс-модули НЕ прокачиваются** (+50%/+25% к ним неприменим): каждое устройство
   работает по-своему, «на 25% лучше клоак» смысла не имеет. `canUpgrade`/UI-кнопка
   апгрейда гасятся для всех аукс-видов (cloak/drone/mielophone/ecm/bomb/scoop).

## Что расходится с текущим кодом (важно при реализации)

- **Класс сейчас у каждого слота** (`Slot.maxClass`, `Hardpoint.maxClass`), а не у корпуса.
- **Лазеры/ракеты рендерятся в точках `hardpoints`** (offset в связанных осях). «Не
  различаем» — только про UI и установку; в 3D стволам/пускам нужны позиции. Оставляем
  `hardpoints` как геометрические якоря; поверх — счётная модель категории.
- **ECM/бомба — врождённые** (`controller.wantsBomb`/`wantsEcm`, `config/weapons.ts`
  BOMB/ECM, `domain/combat/ecm.ts`). **Скуп** — пока только `scoopable` у звёзд, модуля нет.
- **Дрон-отсек** (`DRONE_BAY`, `kind:'drone'`) висит на оружейном пилоне → переедет в аукс.
- **Миелофон** — свой слот только у «Авроры» (`chassis.ts` mielophone slot) → в аукс.
- **Груз уже бесклассовый** (class=1 у всех, любой слот берёт любой контейнер).

## Классы корпусов (безопасные значения — ≥ макс. класса заводских модулей)

Заводские макс-классы: CLOAK_FIELD=3, ENGINE/RCS/SHIELD_STANDARD/BURST_LASER=2, прочее ≤1.

- AURORA_MK3 (игрок, есть CLOAK_FIELD) → **3**
- ATHENA (есть CLOAK_FIELD) → **3**
- APOLLO, ARTEMIS → **3**
- SIDEWINDER/«Арес» (RCS_STANDARD=2) → **2**
- LARGE_FREIGHTER → **3**
- DRONE/«Каллиопа» (всё class 1) → **1**

## Этапы (каждый — сборка + зелёные тесты, отдельный коммит)

1. **Класс корпуса.** `Chassis.class: 1|2|3|4`; установка гейтит по нему, а не по
   `Slot.maxClass`/`Hardpoint.maxClass` (груз-исключение). Правка `shop.ts`
   (`fittingSlots`/`slotsForChassis`/`canFit`/`canBuy`), `shipyard.ts` (`fitOntoChassis`).
   Тесты на «class-too-large» переписать под класс корпуса. ← ПЕРВЫЙ, безопасный.
2. **Категория «аукс».** Новый вид слота, принимающий {cloak, drone, mielophone,
   ecm, bomb, scoop}. Новые модули ECM/бомба/скуп в каталоге. Миелофон — в аукс
   (дрон переезжает вместе с ракетной переработкой, этап 3). Абилки ECM/бомбы гейтятся
   наличием модуля — ДЛЯ ВСЕХ (игрок и ИИ, принцип «неотличимы»): значит ECM/бомбу
   добавить в те ИИ-лоадауты, где они были врождёнными, иначе поведение пропадёт.
   ⚠️ ТОНКОСТЬ: занятость аукс-ячеек считать ПО КАТЕГОРИИ, а не по виду модуля.
   Сейчас примерка считает `installedOfKind` против числа слотов вида — для аукса
   (в одной категории разные виды) это неверно. Ввести `slotCategoryOf(kind)` и
   `installedOfCategory`; поправить `loadout.canInstallInternal`,
   `shop.fittingSlots/slotsForChassis/moduleToReplace/canBuy`, `shipyard.fitOntoChassis`.
   Покрыть тестом: 2 аукс-ячейки, стоит клоак → бомба видит 1 свободную, не 2.
3. **Ракеты — один слот, один тип, боезапас.** Свести пилоны к одной ракетной ячейке;
   render пусков — по сохранённым точкам. Дрон уходит в аукс (см. этап 2).
4. **Ввод: родные клавиши.** Каждое активное аукс-устройство жмётся своей привычной
   клавишей (как сейчас) — только теперь срабатывает лишь при наличии модуля (гейт из
   этапа 2). Единую E НЕ делаем; бинды и описание клавиш восстановить/оставить.
5. **Экран верфи.** Сетка КАРТОЧЕК ОДНОГО РАЗМЕРА — по одной на КАТЕГОРИЮ (двигатель,
   маневровые, щит, гиперпривод, лазеры, ракеты, броня, груз, аукс). Карточка одного
   размера независимо от числа под-слотов. ВНУТРИ карточки — 1–3 под-слота под
   конкретные модули:
   - синглы (двигатель/RCS/щит/гиперпривод/ракеты) — 1 под-слот;
   - лазеры 1–3 (любые, без лево/право), броня 1–3, груз 1–2 (однотипные контейнеры),
     аукс 1–4 (разные устройства).
   Под-слот: пусто = «нет» (тускло), занят — ярко + КРАТКОЕ имя. Клик по под-слоту →
   выбрать/заменить/снять, а на станции — купить (переиспользуем текущий SlotModal-поток).
   У груза дополнительно сводка суммарного тоннажа. Просмотр чужих корпусов на верфи
   показывает те же карточки с пустыми под-слотами (сколько и каких у рамы).
6. **Сброс сейвов** (бамп версии схемы) + чистка. Проверить перенос обвеса при смене корпуса.

## Инварианты для тестов (домен без рендера)

- Установка модуля класса > класса корпуса — отказ (кроме груза).
- Груз влезает в любой корпус независимо от класса.
- Аукс-ячейка принимает cloak/drone/mielophone/ecm/bomb/scoop и отвергает прочее.
- Ракетная ячейка держит один тип; смена типа вытесняет прежний.
- Способность (бомба/ECM) недоступна без соответствующего аукс-модуля.
- Миелофон, стоящий в аукс любого корпуса, разрешает рост (`hasMielophone`).
- Аукс-модуль нельзя прокачать: `canUpgrade` для него — отказ.
