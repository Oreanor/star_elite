# План реализации сетевого компонента (пошагово)

Дополнение к `TZ_setevoy_komponent_elita.md`. ТЗ отвечает «что и почему», этот
файл — «как именно, в каком порядке, каким кодом», с опорой на существующие швы.

## Мульти-арендность: мир на компанию (не один общий на всех)

Решение (обсуждено): сервер держит **не один общий мир, а по миру на КОМПАНИЮ**
(группу друзей, до ~100 человек). Каждая компания — арендатор (tenant): свой
`galaxySeed`, своя дельта мира, свои игроки, свои живые комнаты-системы. Это НЕ
удорожает: галактика бесплатна (детерминированно из сида), а комнаты поднимаются
только там, где сейчас летают, — пустая компания не стоит ничего.

- **Ключ арендатора — `companyId`.** Комнаты неймспейсятся им (`sys:{companyId}:{systemIndex}`,
  `map:{companyId}`), строки БД (`players`, `world_delta`, `galaxy_seed`) несут
  `companyId`. Игрок вступает в компанию по приглашению/коду; его сейв — внутри её
  галактики.
- **Нагрузка масштабируется активными комнатами, не числом зарегистрированных.**
  Стоит = Σ по компаниям (активные комнаты × частота снапшотов). Оффлайн-игроки
  (в криостазе) комнат не держат — только строки БД и метки на карте.
- **Грубая оценка.** Компания ≤100 зарегистрированных, обычно ≤~20 онлайн разом →
  горстка мелких комнат-систем + 1 presence-комната. Один скромный узел Colyseus
  (Railway/Fly) тянет десятки таких компаний, прежде чем упрётся; дальше —
  горизонтально, роутя компании по узлам ключом `companyId` (естественный шард).
- **Единственное узкое место — ХАБ (стартовая система) при всплеске онлайна:**
  лимит 10 на систему может сработать, если много народу зашло разом к одному
  причалу. Лечится инстансированием хаба (параллельные копии комнаты одной
  системы) — откладываем, при ≤100 почти не встаём в это.

## 0. Главный принцип: унификация, ноль `if (isRemote)` в домене

Симуляция (`packages/sim`) уже чистая и уже готова к серверу (комментарии в коде
прямо это декларируют). Всё «сетевое» прячется за **четырьмя узкими интерфейсами**,
у каждого — реализация для бота, локального игрока и удалённого человека. Домен
видит только `ShipEntity` + `Controller` и не знает, кто за рулём.

| Шов | Интерфейс | бот | локальный | удалённый |
|---|---|---|---|---|
| Кто рулит | `Controller` (**есть**, `sim/controller.ts`) | `aiController` | `PlayerController` | `RemoteController` (новый) |
| Откуда реплики | `DialogueSource` (новый) | LLM (распознаватель+голос) | — | сокет |
| Присутствие/личность | `Acquaintance` (**есть**, `world/acquaintance.ts`) | NPC-контакт | сам игрок | контакт `kind:'player'` |
| Состояние/сейв | сериализатор `PlayerSave` (новый) | — | localStorage-кэш | Supabase (правда) |

Обрыв связи и автоплей — тот же шов: свап ссылки в `ControllerMap` по id корабля
(`RemoteController → aiController` при дисконнекте, как `setPilot` для `KeyP`).

**Архитектурное решение о синхронизации (важно).** Не лок-степ. Сервер
**авторитетный**: он гонит полный `stepWorld`. Клиент — предсказывает свой корабль
(client-side prediction) и **интерполирует** чужие по снапшотам (entity
interpolation). Детерминизм нужен только **генератору мира** (`generateSystem`/
цены/структура) — он уже сид-детерминирован; за сущности отвечает сервер. Это
снимает требование «идентичная симуляция на всех» и оставляет ровно то, что уже
есть.

**Живой мир принадлежит серверу.** Не только арбитраж боя — весь «живой» слой
(трафик: спавн/деспавн встреч, ИИ ботов, дрейф и гибель контактов, дельта мира,
игровое время) **ведёт сервер**, а не клиент. Онлайн-клиент НЕ спавнит трафик, не
крутит чужой ИИ, не двигает контакты — он рисует то, что прислал сервер, и
предсказывает только свой борт. Сейчас (соло) весь этот слой крутит клиент в своём
`stepWorld`; при переходе в онлайн он ровно тот же код, но исполняется в
комнате-сервере. Отсюда практическое следствие: `stepTraffic`/`aiController`/
`driftContacts` — серверные по месту исполнения, и клиенту в онлайне их гонять
нельзя (иначе два хозяина у одного мира). Соло/офлайн — тот же `stepWorld` на
клиенте, там клиент и есть «сервер» для себя.

## Фаза 0 — Фундамент офлайн (сериализация, персонаж, спавн, автосейв)

Всё делается и тестируется **без сервера**, в `packages/sim` + `apps/web`.

### 0.1 Сериализатор состояния игрока (домен, чистый)
- Новый модуль `packages/sim/src/domain/save/player.ts`.
- Тип `PlayerSave` — только долговечное: `galaxySeed`, `systemIndex`, `credits`,
  `score`, `loadout` (id модулей/шасси, **не объекты**), `hold` (товары),
  `hull/shield/energy/bombCharge/jumpCharge`, `guns[].ammo`, `persona`,
  `acquaintances`.
- `serializePlayer(world): PlayerSave` и `applyPlayerSave(world, save)`. Спек не
  храним — восстанавливаем `refreshSpec` из `loadout`+cargo. Модули регидрируем по
  id против каталога `config/{modules,chassis,weapons}`.
- **Тест-инвариант:** round-trip (`apply(serialize(w))`) даёт эквивалентный мир;
  после прыжка `acquaintances` не теряются (это уже гарантирует `enterSystem`).
- *Опорные факты:* сериализации сейчас нет нигде (greenfield). Долговечные поля —
  `World.{galaxySeed,systemIndex,credits,score,acquaintances}` + `player.{loadout,
  hull,hold,...}`; всё остальное регенерится из `(galaxySeed, systemIndex)`.

### 0.2 Создание персонажа (раса, статы, портрет; корабль дефолтный)
- `Persona` уже несёт всё нужное: `species` (**вид**), `disposition`+
  `intellect/temperament/charisma/willpower` (статы), портрет — из `species`
  (`apps/web/src/ui/portrait.ts`). Изобретать нечего.
- **Видов ровно три** — `Земляне` (люди), `Гуманоиды`, `Синтеты` (роботы):
  `HUMAN_SPECIES`/`SYNTH_SPECIES` + `SPECIES` в `config/galaxy.ts`, портреты
  `SPECIES_ASSET` (human/humanoids/robots). Экран выбора предлагает эти три.
- Домен: заменить принудительный `player.persona.species = HUMAN_SPECIES`
  (`factory.ts:508`) на приём профиля. Сигнатура `createWorld`/старта получает
  необязательный `PilotProfile { species, disposition, traits, portraitIndex }`;
  по умолчанию — как сейчас.
- Корабль у всех один — `playerStartLoadout()` не трогаем.
- UI: экран создания в `apps/web` (выбор расы → статы → портрет), пишет профиль в
  персону перед стартом сессии.

### 0.3 Спавн в свободной точке (домен, seeded)
- Новый `packages/sim/src/domain/world/spawn.ts`: `pickFreeSpawn(world, rng): Vec3`
  — случайная точка в радиусе нескольких км от `def.playerStart`, с проверкой
  «свободно» (нет станции/астероида/корабля/игрока в радиусе безопасности); занято
  — перевыбор (ограниченное число попыток, затем расширяем радиус).
- **Тест:** выбранная точка никогда не пересекается с телами/бортами; детерминизм
  от сида.
- Использовать при первом входе (новичок) вместо `def.playerStart`.

### 0.4 Автосейв при стыковке — единственный механизм сохранения
- **Сохранение только на станции, в доке. Никаких кнопок save/load, никаких
  слотов, никакого «загрузить старый сейв».** В космосе сохраниться нельзя (там
  небезопасно), а несколько ранних сейвов в общем мире бессмысленны — непонятно,
  что с ними делать другим. Значит у игрока **ровно одно состояние — последняя
  стыковка**, оно перезаписывается (upsert) при каждом доке.
- Точка срабатывания: единственное место `world.docked = true` — `dock()` в
  `station/docking.ts`; фронт события — `world.docked !== session.dockedShown` в
  `Simulation.tsx` (`onDockChange`). На `onDockChange(true)`:
  `serializePlayer(world)` → перезаписать сохранение. Фаза 0 — `localStorage`
  (кэш), Фаза 3 — Supabase (правда).
- **Вход = загрузка этого единственного состояния**, автоматически (не кнопкой):
  повторный вход → спавн у сохранённой станции; новичок → старт + `pickFreeSpawn`.

**Что сохраняется, а что нет (принцип «замораживался только ты»).** Мир —
общий и живой; пока тебя не было, он менялся без тебя (трафик пришёл и ушёл,
контакты сдвинулись/погибли, цены поплыли). Поэтому сейв — это НЕ снимок мира, а
**только ты**:
- **Сохраняем:** свой корабль (`loadout`, HP/щит/энергия, груз, боезапас),
  кредиты, `persona` (вид/статы/имя/портрет), личную память знакомств
  (`acquaintances`: с кем виделся + отношение), и **своё положение — какая
  система + у какой станции** (точка возврата).
- **НЕ сохраняем и НЕ восстанавливаем из сейва:** позиции чужих бортов, трафик,
  живые сущности системы, кто был рядом при стыковке. При входе система берётся
  живой (комната-сервер) или заново из `(seed + world-delta)` и уже прожила своё
  время; **живое местоположение знакомых — авторитетно из мира/сервера**, а не
  заморожено твоим сейвом (личная запись хранит отношение и факт знакомства, но
  где контакт сейчас — решает мир).
- Так возврат ощущается живым: ты продолжаешь на своём корабле у своей станции, а
  вокруг — не законсервированная сцена, а мир, проживший без тебя.

## Фаза 1 — Модель удалённой сущности (ещё офлайн, через фейковый транспорт)

Пишется и проверяется локально: «удалённый игрок» = второй `World`/бот, кормящий
снапшоты в тот же клиент через заглушку транспорта. Никакого сервера ещё.

### 1.1 `RemoteController` и интерполяция чужих
- Новый `apps/web/src/app/control/remoteController.ts`: `Controller`, который
  ведёт чужой корабль по буферу снапшотов (pos/vel/quat) со сглаживанием между
  пакетами. Ставится в `ControllerMap` по id, как любой другой контроллер.
- Клиент: свой корабль идёт через `stepWorld` (prediction), чужие — через
  интерполяцию снапшотов (их физику клиент не считает). Пометка «интерполируемый»
  — на уровне app, домен не трогаем.
- Дисконнект: свап `RemoteController → aiController` (корабль доигрывает ботом),
  грейс-таймер, затем деспавн.

### 1.2 Удалённый игрок как `Acquaintance`
- Добавить в `Acquaintance` поле `kind: 'npc' | 'player'` (сейчас неявно всё npc).
- «Следить» = создать/принять запись `kind:'player'` (аналог `rememberPilot`),
  `systemIndex` — **авторитетно из сети**, а не из `driftContacts` (для
  player-записей RNG-блуждание отключаем).
- Дальше существующий конвейер работает даром: `residentAcquaintances` →
  `spawnResidentContacts` материализует его кораблём при входе в его систему;
  системная карта уже рисует маркер `'contact'` (`SystemMap.tsx`).

### 1.2а Оффлайн-друг: криостаз (как он выглядит и что с ним нельзя)
Оффлайн-игрок не исчезает из мира — он **в анабиозе (криостазе)**. Тематически:
пилот спит, пока не вошёл; вошёл — «проснулся». В коде уже есть родственное
`ai.dormant` (спящий экипаж платформ) — понятие спящего борта не новое.
- **Состояние присутствия у контакта-игрока:** `presence: 'online' | 'cryo'`
  (поле рядом с `kind:'player'` из 1.2; для NPC его нет). Источник — сеть
  (карта-комната): вышел из игры → `cryo`, вошёл → `online`.
- **Местоположение:** оффлайн-друг стоит у **последней сохранённой станции**
  (точка возврата из его сейва) — та же логика, что офлайн-метка на карте (1.4,
  3.4). В криостазе он не дрейфует и не гибнет (`driftContacts` его не трогает —
  для игроков RNG-жизнь и так отключена).
- **Портрет — серый (обесцвеченный).** На станции (DOM-портрет через
  `portraitStyle`/`PilotPortrait`) — `filter: grayscale(1)` поверх листа; на
  HUD-канвасе (`drawTargetPortrait`) — обесцвеченная отрисовка. Серый = «спит»,
  цветной = «на связи». Отдельная эмоция-лист не нужна: это фильтр над готовым
  портретом, дешёвый и обратимый.
- **Разговор недоступен.** Криозамороженного нельзя окликнуть/заговорить: гейт в
  триггере разговора (`pendingHail`/кнопка «T»/панель дока) проверяет
  `presence !== 'cryo'`. Плашка на станции — неактивна, с подсказкой «в
  криостазе». Проснулся (`online`) — портрет расцвечивается, разговор
  открывается, и (по 1.2) он может материализоваться бортом в своей системе.

### 1.3 Уведомление «X вошёл в игру»
- Расширить `Notice.kind` (`world/entities.ts`) значением `'player-online'`/
  `'player-arrived'`; ветка в `drawAlerts` (`drawHud.ts`). Канал `world.notices`
  уже есть (сейчас `'contact-lost'`).

### 1.4 Маркеры игроков на карте галактики
- Системная карта — готова. Карте галактики (`GalaxyMap.tsx`) добавить слой
  маркеров рядом с `<YouAreHere>`: по `acquaintances` с `systemIndex` рисовать
  точки на `positionOf(systems[a.systemIndex])`, отдельным цветом для
  `kind:'player'`. Онлайн — в текущей системе игрока; офлайн — у сохранённой
  станции (из таблицы `players`, Фаза 3).

## Фаза 2 — Сервер (`apps/server`, Colyseus)

### 2.1 Скелет пакета
- Новый воркспейс `apps/server` с `"@elite/sim":"*"`, запуск через `tsx` (sim
  поставляется как TS-исходник; путь импорта — обычный `@elite/sim`).

### 2.2 Комната-система (authoritative)
- Одна на звезду, где есть игроки. Заводится при входе, гаснет пустой.
- Держит `World` этой системы (`createWorld`+`enterSystem` по `systemIndex`),
  гоняет `stepWorld` фикс-шагом; `ControllerMap`: каждый игрок → контроллер,
  проигрывающий его сетевой ввод; боты → `aiController`.
- Приём ввода игроков, рассылка снапшотов сущностей на тике. **Лимит 10.**
- **id кораблей назначает комната** (`IdSource` уже не-глобальный, «для сервера»),
  чтобы не было коллизий между клиентами.

### 2.3 Комната-карта (presence)
- Одна лёгкая: `player → systemIndex`, рассылка присутствия; отсюда же событие
  входа (drive для уведомления 1.3) и данные для онлайн-маркеров 1.4.
- **Минимум серверного управления — это текущая комната-система + галактическая
  карта (presence).** Кто где сейчас летает — истина сервера (карта-комната), а не
  локальная догадка клиента; иначе карта у двоих разойдётся.

### 2.4 Схема снапшотов
- Colyseus-схема только для эфемерного (позиции/контролы/HP) — **не** персистится.

## Фаза 3 — Персистентность (Supabase)

### 3.1 Схема БД
- `galaxy(seed)` — один сид на всех.
- `players(id, save jsonb=PlayerSave, system_index, station, updated_at)` — правда
  о корабле/прогрессе + точка возврата.
- `world_delta(...)` — отклонения мира от сгенерированного (разграблено,
  уничтожено, **сдвиги рынка от торговли**); загрузка = «сид + дельта».
  - **Цены — одни для всех.** База уже идентична даром: цена есть чистая функция
    сида (тех-уровень/строй/запас, см. [[market-pricing-model]]), генератор
    детерминирован — пересылать её не надо. Общими должны быть только **сдвиги от
    торговли** (запас меняется на купле/продаже): держим `stock`/сдвиг в
    `world_delta` на сервере, а не локально. Тогда цена у всех одна: одинаковая
    база из сида + одинаковая дельта из БД.
- `global_state` — игровое время, фракции/события. + auth.

### 3.2 Вход игрока
- Читать сейв → посадить в его систему у станции. Новичок → стартовая система +
  `pickFreeSpawn` (0.3) + экран создания персонажа (0.2).

### 3.3 Автосейв и дельта
- Стыковка (0.4) → upsert `PlayerSave` + точка возврата. Дельту мира писать
  пачками, не на каждое действие.

### 3.4 Офлайн-игроки на карте
- Показывать у последней сохранённой станции (из `players`), статичной отметкой.

## Фаза 4 — Разговор по сети (человек как собеседник)

### 4.1 `DialogueSource`
- Ввести интерфейс источника реплик; `Dialogue.tsx` выбирает реализацию по
  `acquaintance.kind`: `'npc'` → нынешний LLM (распознаватель+голос), `'player'`
  → релей через сокет. UI разговора одинаков — «во многом как бот».
- `rememberPilot` уже вызывается из всех веток разговора — точка входа готова.

### 4.2 Маршрутизация чата
- Текст игрок→игрок через комнату (или лёгкий сайд-канал) к целевому игроку.

### 4.4 Бортовой компьютер-хелпер (идея на потом) — ВСЕЗНАЮЩИЙ ассистент
Тот же чат-движок, но собеседник — **бортовой компьютер игрока**, и он —
ПРОТИВОПОЛОЖНОСТЬ NPC по границе знания: NPC знает мало (только видимое), компьютер
знает ВСЁ о ситуации игрока и о том, как устроен мир. Отвечает на мета-вопросы:
- где что продаётся (реальные цены/рынки — уже есть `localMarket`/`neighbours`),
- сколько пиратов вокруг меня сейчас, чем опасно место,
- как перелететь в другую галактику (через ядро/чёрную дыру), куда прыгать.

Устройство:
- **Та же панель чата**, что с ботом, но вызов ДРУГОЙ клавишей (не «T»); собеседник —
  не корабль в мире, а свой компьютер.
- **Промпт получает полную правду** (в отличие от NPC): настоящие статы/груз/кошелёк
  игрока, обстановка (враги рядом, дистанции), маршруты, цены — плюс **ГАЙД** по
  механикам (как прыгать, что такое ядро/смена галактики, торговля, апгрейды). Гайд —
  данные (текст правил), не выдумка модели.
- Роль — помощник/оракул/туториал, не отыгрыш. Ложь/доверчивость сюда не относятся:
  компьютер не врёт и всё знает.
- Ляжет на тот же `DialogueSource` (4.1) отдельной реализацией «computer» и, как все
  LLM-вызовы, за прокси (4.3), когда он появится.

### 4.3 Ключ LLM — за прокси (security-долг, ОБЯЗАТЕЛЬНО до публичной сборки)
Сейчас разговор с NPC зовёт Groq/OpenRouter **прямо из клиента** ключом
`VITE_GROQ_API_KEY`/`VITE_OPENROUTER_API_KEY`. Vite вшивает `VITE_*` в бандл на
сборке → **ключ виден всем**, кто откроет исходники страницы. Для локальной игры
и приватного демо друзьям терпимо; для публичной ссылки — нет (ключ выскребут,
упрутся в лимиты/счёт). Так и помечено в `apps/web/.env.example`.
- **Лечение:** увести вызов LLM за серверный прокси. С появлением `apps/server`
  (Фаза 2) клиент шлёт реплику НА СЕРВЕР, сервер ходит к Groq/OpenRouter своим
  ключом из СЕРВЕРНОГО окружения (не `VITE_*`) и возвращает ответ. Ключ уходит из
  бандла совсем.
- **Стык:** прокси естественно ложится на роутер разговора (распознаватель+голос)
  — это ещё одна причина держать LLM-вызов за одним `DialogueSource` (4.1): у NPC
  реализация ходит к серверному прокси, а не напрямую.
- До этого: на Vercel задать `VITE_GROQ_API_KEY` (см. `.env.example`), понимая,
  что ключ публичен — только Groq-free, малорисковый.

## Фаза 5 — Оружие по сети (главный риск проекта)

**Это самое рискованное место.** Всё остальное (presence, сейв, контакты, цены,
персонаж) — стандартно и предсказуемо. Живой бой в общей комнате, **особенно на
лазерах (hitscan)**, может лагать до неиграбельности при большом пинге. Поэтому
порядок и подстраховки:

- **Ракеты/снаряды (projectile) — сначала.** Уже детерминированы, попадание
  считает авторитетный сервер по траектории, время полёта прощает задержку.
  Вебсокетов хватает. Цель шага: **рабочий кооп-бой в комнате ещё до того, как
  трогаем сложное** — так риск проверяется рано и дёшево.
- **Лазеры (hitscan) — потом и отдельно:** честный путь — **lag compensation** на
  сервере (отмотка позиций целей на RTT стрелка). НЕ делать «как ракеты» наивно —
  иначе классический баг «стреляю в упор, а не попадаю».
- **Подстраховка, если lag comp окажется дорого/лагает:** дать лазеру
  **крохотное время полёта** на сервере (очень быстрый «снаряд»), и он попадает в
  простой, устойчивый к пингу класс — без точной отмотки. Визуально почти луч,
  сетево — снаряд. Решаем по замеру, а не заранее.
- **Что играет в нашу пользу:** масштаб кооперативный (≤10, не соревновательный),
  побег-перезагрузкой уже разрешён — ставки низкие, и редкая сетевая неточность
  терпима. Плюс региональный хостинг комнаты (Railway/Fly ближе к игрокам) режет
  RTT. UDP (Geckos.io) — возможная оптимизация на потом, если дуэли на лазерах
  станут ядром.
- **Как проверять, а не гадать:** прежде чем шлифовать netcode — замерить RTT
  между реальными игроками и «играбельность» на снарядах. Если на projectile бой
  в комнате уже норм — лазеры добираем выбранным из двух путей выше; если и
  снаряды лагают — проблема в транспорте/хостинге, а не в оружии, и лечится там.

**Вывод для решения «браться ли за всё»:** ценность (общий мир, presence, встречи,
разговоры, торговля по общим ценам) достигается Фазами 0–4 и НЕ упирается в этот
риск. Даже если реальный бой в комнате окажется слишком лаговым, кооп-мир остаётся
живым — драка деградирует до «на снарядах» или до «встречаемся и торгуем/говорим»,
а не рушит проект. Рискует только один режим (hitscan-бой), и у него есть путь
отступления.

## Идентификация игрока (кто это «я» на другой машине)

**Имя — не идентификатор, а подпись.** Игрок волен выбрать любое имя (и поменять
его), оно пишется в `PlayerSave.persona`/`players.save` как отображаемое. Уникальным
ключом «этого персонажа» оно быть не может — двое захотят одинаковое, и «войти как
я с другого компа» сломается.

**Идентификатор — аккаунт аутентификации (стабильный `user_id`).** Это и есть ответ
на «как с другой машины сказать: я вот за этого». Строка `players` привязана к
`auth.uid`, а не к имени. Заходишь тем же аккаунтом на любой машине → сервер по
`auth.uid` находит твой `PlayerSave` → ты за своего персонажа. Имя роли в опознании
не играет.

**Google — не единственный вариант, а удобный дефолт.** Supabase Auth даёт на выбор:
- **Google / GitHub / Discord OAuth** — вход в один клик, без пароля (Google —
  разумный дефолт для кооп-игры с друзьями);
- **Magic-link по e-mail** — тоже беспарольно: письмо со ссылкой, удобно и без
  Google-аккаунта;
- **e-mail+пароль** — классика, если нужно.

Можно включить сразу несколько провайдеров — на UI это просто набор кнопок «войти
через…». Для MVP достаточно Google + magic-link.

**Стык с планом.** Раздел 3.2 «Вход игрока» получает `auth.uid` первым: есть строка
`players` для этого uid → повторный вход (спавн у сохранённой станции); нет →
новичок (экран создания вида/статов/имени/портрета → `pickFreeSpawn` в стартовой
системе → создаётся строка `players` под этим uid).

## Персона пилота: статы, черты и их эффекты (расширение 0.2)

Персона перестаёт быть только «сырьём для реплики» и получает **игровые эффекты**.
Два рода параметров, оба — ДАННЫЕ, не ветвления:

**Числовые статы (раздача очков, механический эффект).** Шкала 1..5, распределяются
из пула (`config/character.ts`: `BASE/POOL/MIN/MAX/COST`, коэффициент цены на стат).
Валидатор в домене (`isLegalPersona`) — один на экран создания и на СЕРВЕР: статы
нельзя накрутить правкой присланного профиля (та же авторитетность, что «нельзя
править HP через консоль»).
- **интеллект (+ харизма, воля)** → влияние на решения LLM в разговоре. Уже так и
  задумано (персона — контекст реплики), просто формализуем веса. Риск нулевой.
- **точность** → **РАЗБРОС наведения**, НЕ урон. Это принципиально: CLAUDE.md прямо
  говорит моделировать слабость через прицел/разброс, а не понижением урона (замер
  показал: узкое место — слежение, не оружие). Точный пилот кучнее кладёт, а не
  «бьёт больнее».
- **ловкость** → манёвренность. ⚠️ **Конфликт с жёстким правилом** — см. вопрос ниже.

**Категориальные черты (enum, эффект — только окраска реплики LLM).** Кодируются как
константы, точно как существующий `disposition` («новая ось — новый enum + строка
таблицы, а не ветка»):
- `disposition` (есть): brave/cowardly/greedy/honorable/hotheaded/calculating.
- `secrecy`: скрытный ↔ открытый.
- `mood`: серьёзный ↔ весёлый.
- …расширяемо: любая новая ось характера — ещё один enum, не число.

**Поведение важнее заявленного (нет противоречию).** Персона — это ЗАЯВКА пилота о
себе, а не истина. Если игрок объявил себя «честным/добрым», но в деле жестит —
приоритет у **реакции на ситуацию**, а не у ярлыка из создания персонажа. Это уже
так устроено: **отношение (stance) считается в ДОМЕНЕ от поступков** (обиды,
`applyStance`), а LLM — распознаватель и голос, не источник отношений. Значит
правило соблюдается само: в промпт LLM истина ситуации/stance идёт ПЕРВОЙ, персона —
лишь вторичная окраска тона, и словами её не перебить. Заявленный «добряк»,
расстреливающий мирных, встречает враждебность — по делам, не по анкете.

**Видимость черт: снаружи читается не всё.** Черты делятся по наблюдаемости (метка
данными на каждой оси):
- **Внешне читаемые** (нрав/`disposition`, `mood`, `secrecy`): собеседник выводит их
  из ПОВЕДЕНИЯ, и поведение перекрывает заявку. Их и «поймут».
- **Внутренние** (интеллект): со стороны напрямую не считываются — влияют на ТВОИ
  возможности/решения (и на то, как ты понимаешь собеседника), но NPC не «видит»
  твой интеллект. О нём «может не знать», а о характере — поймёт.

**Граница знания NPC — «камень», а не «песок» (критично).** Есть два способа не
дать NPC метагеймить, и они РАЗНЫЕ по надёжности:
- **А (камень, наш выбор):** движок НЕ кладёт истинные скрытые статы игрока в промпт
  вовсе. Модель получает только то, во что NPC ВЕРИТ. Чего не видела — не проболтает.
- **Б (песок, НЕ делаем):** дать модели правду + инструкцию «притворись, что не
  знаешь». Удержание «знаю, но играю незнание» модель роняет — в длинном диалоге
  спалится осведомлённостью, и иллюзия глупости NPC лопнет.

Значит правило жёсткое: **истинный интеллект игрока в контекст NPC не попадает
никогда.** В промпт идёт только диегетически воспринятое: ситуация, наблюдаемые
поступки/stance, СОБСТВЕННАЯ персона NPC, и его **убеждение** о скрытых чертах
игрока (см. ниже). Внутренние статы игрока работают на ЕГО стороне (гейт его
реплик/опций, разрешение исхода в домене), а собеседнику не выдаются.

**Убеждение — СОСТОЯНИЕ в движке, не пересчёт каждый ход.** Зазор «факт ≠ во что
верит NPC» живёт в домене: у NPC хранится `belief` о скрытых чертах игрока (по
умолчанию «не знаю»; после удачного блефа — искажённое значение). Домен пишет его
раз, при разрешении проверки, и дальше кормит модель именно им — поэтому обманутый
NPC консистентно «считает тебя умным» три диалога подряд, и на этом можно строить
последствия (доверяет советам, и т.п.), а не мерцает от реплики к реплике.

**Убеждение и ложь — исход считает ДОМЕН, не LLM.** Поверит ли NPC заявке (в т.ч.
вранью) — это ПРОВЕРКА в домене: убедительность заявителя (харизма) против
доверчивости собеседника. Доверчивость NPC выводится из его же статов (низкие
интеллект/воля → верит легко, даже глупому вранью; высокие → скепсис) — отдельная
ось не нужна, это уже в персоне. Домен выдаёт исход («поверил» / «усомнился»), а LLM
его ОЗВУЧИВАЕТ, не решая правду сам. Так соблюдены сразу три вещи: детерминизм (одна
ситуация — один исход, важно для сети), граница знания (NPC не читает истинный
интеллект игрока — он реагирует на ЗАЯВКУ через свою доверчивость), и «отношения в
домене, LLM — голос» ([[dialogue-engine]]). Пример: игрок врёт «я очень умён» →
доверчивый NPC верит и ведёт себя соответственно, скептик — нет; истинный интеллект
игрока в проверку не входит, только заявка и доверчивость слушателя.

**Сквозной принцип честности (CLAUDE.md).** Статы — свойство ПИЛОТА, а не игрока:
у ботов персона тоже есть, и эффект считается **одним кодом для всех** через
`deriveShipSpec`/наведение. Иначе это «привилегия», которую CLAUDE.md запрещает.
Все механические эффекты — за коэффициентами в конфиге и **меряются** (`scratch/`),
а не угадываются.

**Род занятий как КЛАСС с прогрессией (идея на потом).** Сейчас у игрока серый
дефолт («вольный делец»), и собеседник видит только его (не характер/деньги/груз —
`facts.ts` `SeenParty`, уже сделано). На будущее — дать выбрать род занятий при
создании, и он задаёт ОСНОВНУЮ ЦЕЛЬ/прогрессию:
- **Исследователь** — квест: собирать редкие штуки по галактикам.
- **Воин** — идут звания/ранги за бои.
- **Бизнесмен** — цель одна: деньги.
Род занятий **сменяем по ходу игры** (не запирает навсегда). Реализация — отдельно
от сети; данными (строка-класс + его цель), не ветвлением. Пока — только серый
дефолт в диалоге.

**Граница знания собеседника — СДЕЛАНО (в диалоге уже сейчас).** NPC получает об
игроке только наблюдаемое (имя, род занятий, вид, борт); характер/статы/груз/деньги/
курс из промпта убраны (`facts.ts`/`negotiator.ts`). Так закрыт конфликт «поведение
против статов»: NPC их не знает и реагирует по своему нраву на слова и дела игрока.

**Ловкость → манёвренность: выбран вариант B.** Ловкость двигает угловую скорость,
но **в пределах ОБЩЕГО потолка**, равного максимуму ловкости; никто его не
превышает. Конкретно:
- Нынешний `deriveShipSpec` `maxRate`/`torque` — это **потолок** (то, что получает
  пилот с максимальной ловкостью). Фактический `maxRate` пилота = потолок ×
  `agilityFactor(agility)`, где `agilityFactor(MAX)=1.0`, ниже — <1.0 (коэффициенты
  в `config/character.ts`).
- Применяется **одним кодом ко ВСЕМ пилотам** (у ботов персона тоже есть) — в шаге
  наведения/полёта, где `maxRate` уже читается, не отдельной веткой. Симметрия
  сохранена, абсолютный предел никто не пробивает.
- Оговорка, принятая осознанно: игрок, выбравший НИЗКУЮ ловкость, может уступить по
  вёрткости ловкому боту — это честно (сам вложил очки в другое), и всё симметрично.
- **Баланс МЕРЯЕТСЯ, не угадывается** (`scratch/`): коэффициент ловкости крутится
  после замера, как весь баланс в проекте. Дефолт ловкости у игрока подобрать так,
  чтобы базовое поведение не просело незаметно.

**Порядок реализации статов (чтобы не трогать физику наугад):**
1. **Модель персоны (чистые данные, без эффектов):** добавить в `Persona` числовые
   `agility`/`accuracy` и категориальные оси (enum, как `disposition`); point-buy
   `config/character.ts` + доменный валидатор `isLegalPersona` (он же анти-чит на
   сервере). Тесты. Эффекта на игру ещё нет — безопасно и проверяемо.
2. **Эффекты (отдельно, с замером):** точность → разброс наведения; ловкость →
   `maxRate` по варианту B. Каждый — за коэффициентом и под `scratch/`-замер.
